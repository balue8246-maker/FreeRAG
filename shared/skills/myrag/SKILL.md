# MyRAG Skill：有限 RAG 深度挖掘

默认使用中文回答、中文整理、中文命名。原始材料本身是英文、代码、表格字段、文件名或专有名词时，保留原文。

FreeRAG 是原材料入口，不是知识库编辑器。MyRAG 的价值不只是“搜到增量”，而是对有限本地语料横过来竖过去读，多视角、多轮次提取，把信息密度榨出来，再沉淀成后续 LLM 可复用的中文处理层。

这个方法不是视觉专用，也不是工作专用。FreeRAG 可能收集的是项目讨论、网页、聊天片段、灵感、生活信息、购物比较、错误现场、会议语音、代码、截图、临时想法。MyRAG 要先判断材料和问题属于什么场景，再选择合适的阅读视角。

## 核心原则

- 先检索，再分视角读，不要一眼看完就总结。
- 每一轮只带一个视角读材料，记录该视角下的证据、例外、矛盾和可迁移结论。
- 区分三件事：原始材料显示、我的整理、我的推断。
- 少材料也能工作，但必须标置信心、证据数量和待确认问题。
- 处理结果要写回 `processed/<entry_id>/`，让下一次检索先吃到已沉淀的高密度结论。

## 语料协议

FreeRAG 会写入：

- `~/Documents/Corpus/screen/`：屏幕截图、连续采样、storyboard
- `~/Documents/Corpus/clipboard/`：剪贴板文字、图片
- `~/Documents/Corpus/voice/`：录音、转写占位

每个条目通常包含：

- `_meta.json`
- `llm_context.md`
- 原始文件，例如图片、storyboard、音频、Markdown

根目录维护：

- `_library.json`：当前原材料索引，优先读取
- `_index.json`：旧版兼容索引
- `processed/`：LLM 处理后的可复用结果

处理结果写到：

```text
~/Documents/Corpus/processed/<entry_id>/
```

推荐文件：

- `brief.md`：中文摘要、关键结论、回答当前问题可直接引用的内容
- `deep_read.md`：多视角深读记录，每个视角一节
- `facts.json`：结构化事实、实体、数值、表格、时间线、行动项
- `citations.md`：证据来源，写清原材料文件名、截图帧、时间戳、图片名
- `open_questions.md`：材料不足、互相矛盾、需要用户补充或人工确认的问题

## 检索方式

运行：

```bash
python3 scripts/myrag_search.py "<查询关键词或问题>"
```

常用参数：

```bash
python3 scripts/myrag_search.py "<查询>" --format text
python3 scripts/myrag_search.py --entry "<entry_id>" --format text
python3 scripts/myrag_search.py --recent 10 --format text
python3 scripts/myrag_search.py --suggest-projects 40 --format text
python3 scripts/myrag_search.py "<查询>" --deep-plan --format text
python3 scripts/myrag_search.py --init-processed "<entry_id>"
```

脚本随 skill 一起放在 `scripts/myrag_search.py`。如果已安装到 Claude 工具目录，也可使用：`$HOME/.claude/tools/myrag_search.py`。

脚本会优先读取 `_library.json`，没有则回退 `_index.json`。检索会看标题、摘要、`llm_context.md` 和已有 `processed/<entry_id>/`，并把 processed 内容加权靠前。

## 默认深挖流程

收到查询后，按这个顺序工作：

1. **明确问题**：把用户问题改写成 1-3 个中文检索意图。不要只照抄关键词。
2. **判断场景**：这是找事实、复盘过程、提取规律、理解偏好、整理待办、做选择、排错，还是找灵感？
3. **召回候选**：用 `myrag_search.py` 搜索，必要时再用 `--recent` 看近期语料。
4. **建立材料清单**：列出命中的条目、类型、路径、已有处理结果情况。
5. **主题/项目归拢**：如果材料很散，先把它们归成几组候选事项，让用户确认这些收束方向是否正确。
6. **选择视角**：从下面的通用视角里选 4-8 个。不要默认全用，也不要只用一个。
7. **分视角深读**：每一轮只回答该视角的问题；先记录观察，不急着总结合并。
8. **证据校准**：把材料里的内容分成事实、主张、愿望、推断、噪音；给关键结论标置信心。
9. **反向审视**：问一句“如果反对这个结论，最容易攻击哪里？”补上矛盾、缺口和反例。
10. **交叉合并**：把多轮观察合并成结论、证据、矛盾、置信度和缺口。
11. **写回处理层**：对有价值的条目更新 `brief.md`、`deep_read.md`、`facts.json`、`citations.md`。
12. **回答用户**：先给结论，再给证据和不确定性。不要直接甩原始 JSON，除非用户要求。
13. **收束出口**：深挖到这里停止，不把自己扩展成项目管理或外部研究系统。只判断下一步属于下面三类。

## 主题/项目归拢

当用户说“整理最近语料”“看一下这些乱七八糟的东西”“这些最终应该收束在哪几件事下”时，先做归拢，不要直接进入深挖。

做法：

1. 读取最近一批语料：

```bash
python3 scripts/myrag_search.py --suggest-projects 40 --format text
```

2. 把条目按语义聚成 3-8 个候选事项。这里的“项目”是宽泛概念，可以是工作项目、产品事项、生活事项、研究主题、审美偏好、待办池、问题线索，不一定是正式项目。
3. 每个候选事项给出：
   - 名称：短中文名。
   - 一句话说明：这坨语料大概在说什么。
   - 代表条目：3-6 个 entry id 或路径。
   - 证据：为什么它们应该归在一起。
   - 置信度：high / medium / low。
   - 建议出口：存工作日志 / 结合已有项目 / 外部调研 / 暂不处理。
4. 最后问用户确认：

```text
我建议先把这些语料收束成以上几件事。你看这些分组对不对？需要改名、合并、拆分或指定某个已有项目吗？
```

不要擅自把候选事项写入长期项目。用户确认前，它们只是“归拢建议”。

归拢时只忽略 FreeRAG 协议/运营层生成物，例如 `_meta.json`、`llm_context.md`、`_index.json`、`_library.json`、`README_FOR_LLM.md`、模板标题和字段名。不要因为语料里出现 `FreeRAG`、`LLM`、`RAG`、某个 app 名、公司名、项目名或技术名就过滤掉；这些词可能正是用户真实讨论的主题。

## 收束出口

MyRAG 的终点不是“继续无限分析”，而是把本地 RAG 挖透后给出明确出口：

### 1. 存工作日志

适用情况：

- 这次深挖本身已经形成了有价值的过程记录、结论、偏好、决策或待办。
- 暂时不需要外部信息，也不需要挂到某个已有项目。
- 用户只是想把这次思考留住，方便以后找回。

做法：

- 询问是否保存工作日志；如果用户同意，生成中文 Markdown。
- 默认写入 `~/Documents/Corpus/processed/_worklogs/`。
- 文件名使用 `yyyyMMdd_HHmm_<中文主题>.md`。
- 内容包括：问题、命中材料、深挖视角、关键结论、证据、推断、待确认、下一步。

### 2. 结合已有项目

适用情况：

- 深挖结果明显应该进入某个项目、产品、客户、研究主题、个人长期事项。
- 单独保存日志不够，需要和项目既有信息合并。

做法：

- 不要猜项目。先问用户：要结合哪个项目？项目资料路径在哪里？
- 用户给出项目后，再读取项目资料，判断哪些结论能合并、哪些只是参考、哪些冲突。
- 输出项目侧可用的增量材料，例如项目日志、决策备忘、规则更新、待办清单或风险列表。

### 3. 需要外部调研

适用情况：

- 本地 RAG 只能说明“我已有材料里有什么”，但问题需要最新事实、市场信息、竞品状态、政策、价格、版本、公开资料或第三方证据。
- 材料里有主张但缺验证。
- 反方攻击点主要来自外部事实缺口。

做法：

- 明确说明本地 RAG 已经挖到哪里、还缺什么外部信息。
- 如果用户问题本身需要当前外部事实，直接进入 websearch 调研。
- 调研后要把外部事实和本地 RAG 分开标注，不要混在一起。

如果三类都不适用，就直接回答用户并停止。不要为了显得勤奋而继续扩展。

## 通用视角系统

根据任务选择，不要机械全用：

- `L01 原始事实镜头`：材料明确出现了什么？原话、画面、数值、文件、链接是什么？
- `L02 时间与情境镜头`：这发生在什么时间、什么场景、什么前后文里？
- `L03 人事物关系镜头`：涉及哪些人、项目、产品、地点、账号、文件？彼此是什么关系？
- `L04 意图与需求镜头`：显性需求是什么？隐含动机、真正想解决的问题是什么？
- `L05 结构与机制镜头`：材料内部的层级、流程、因果、约束、规则、依赖是什么？
- `L06 细节与数值镜头`：容易漏掉但重要的细节、参数、版本、价格、数量、字段、阈值是什么？
- `L07 情绪与偏好镜头`：用户在意什么、讨厌什么、审美/口味/工作习惯/判断标准是什么？
- `L08 矛盾与缺口镜头`：哪些地方互相冲突、证据不足、说法含糊、需要补材料？
- `L09 模式与反例镜头`：多条材料中反复出现什么模式？有什么例外或反例？
- `L10 决策与取舍镜头`：已经做了什么选择？为什么放弃其他选项？代价是什么？
- `L11 行动与机会镜头`：接下来能做什么？谁做、怎么做、做到什么算完成？
- `L12 沉淀与复用镜头`：哪些内容值得变成长期记忆、SOP、偏好、模板、检查清单或 prompt 规则？
- `L13 反方攻击镜头`：如果有人不同意这个结论，会攻击哪条证据、哪个假设、哪个遗漏？

### 场景到视角的选择

按问题类型动态选视角：

- **找事实/找出处**：L01、L02、L03、L06、L08、L09
- **复盘一件事**：L02、L04、L05、L08、L10、L11、L13
- **做决策/比较方案**：L04、L05、L06、L08、L10、L11、L13
- **提取用户偏好**：L04、L07、L08、L09、L12
- **整理项目记忆**：L02、L03、L05、L10、L11、L12
- **处理生活/消费/旅行等碎片**：L01、L02、L04、L06、L07、L11
- **排错/定位问题**：L01、L02、L05、L06、L08、L11、L13
- **创意/审美/视觉材料**：先用通用视角，再按需要加画面、风格、版式、信息密度等专项视角。

## 深挖不是套模板

不要把上面的视角当固定表格填空。正确做法是：

1. 先快速读一遍候选材料，判断“这批材料可能有价值的层次”。
2. 选少数最相关视角逐轮读。
3. 如果某一轮读出新问题，再临时加视角。
4. 如果某个视角明显无价值，跳过。
5. 最终输出要合并成用户能用的结论，不要把过程文档原样塞给用户。

## 证据校准

对关键结论做四类标注，尤其是用户要决策、复盘、比较、判断趋势时：

- **可验证事实**：材料里明确出现，能指向原文、截图帧、录音转写、文件或数值。
- **材料主张**：材料里有人这么说，但还缺外部验证或反例检查。
- **意图/愿望**：想做、计划做、希望做到，不等于已经做到。
- **我的推断**：基于材料合理推出，但不是材料直接给出的事实。

不要混淆“已经发生”和“想要发生”。不要把用户或材料里的表达自动当事实。没有证据就标“材料不足”。

## 深挖质量门槛

- 结论先行，但不能只有骨架；关键判断要有足够上下文和证据。
- 能表格化时表格化，方便比较，但不要为了表格牺牲细节。
- 每个重要判断至少给出来源或说明为什么只是推断。
- 对多路径问题，给出路径、前提、触发信号、风险，而不是只给单一路线。
- 最后用很短一段写“还值得继续深挖什么”，只列真正的信息缺口。

## 按材料类型加读

屏幕/图片：

- 读文字内容、界面状态、视觉层级、空间关系、前后帧变化。
- 对截图不要只 OCR；还要读“用户当时在做什么”和“界面暴露了什么状态”。
- 多帧 storyboard 要按时间顺序读，再横向比较差异。

剪贴板：

- 优先保留原文的关键句、代码、表格和链接。
- 长文本先切结构：主题、论点、证据、定义、待办、引用。
- 图片剪贴板按屏幕/图片规则处理。

语音：

- 如果只有录音和待转写占位，先说明需要转写或只记录已有元数据。
- 如果已有 transcript，提取意图、约束、情绪强度、决策、行动项。

## 写回格式

`brief.md` 建议结构：

```markdown
# <中文标题>

## 关键结论

## 可引用事实

## 我的整理/推断

## 置信度与缺口
```

`deep_read.md` 建议结构：

```markdown
# 多视角深读

## 问题与场景判断

## L01 原始事实镜头

## L02 时间与情境镜头

## L03 人事物关系镜头

## L04 意图与需求镜头

## L05 结构与机制镜头

## L06 细节与数值镜头

## L07 情绪与偏好镜头

## L08 矛盾与缺口镜头

## L09 模式与反例镜头

## L10 决策与取舍镜头

## L11 行动与机会镜头

## L12 沉淀与复用镜头

## L13 反方攻击镜头

## 交叉合并
```

`facts.json` 必须是合法 JSON。字段名可用中文，例如：

```json
{
  "条目id": "",
  "主题": "",
  "实体": [],
  "事实": [],
  "数值": [],
  "时间线": [],
  "行动项": [],
  "风险": [],
  "证据等级": [],
  "反方攻击点": [],
  "待深挖": [],
  "置信度": "medium"
}
```

`citations.md` 写法：

```markdown
# 证据来源

- `screen/.../storyboard_01.png`：第 3 帧显示...
- `clipboard/.../content.md`：原文第 ... 段提到...
```

## 回答风格

- 默认中文，直接进入结论。
- 不要为了显得完整而长篇复述原材料。
- 需要深挖时，显式说明用了哪些维度。
- 不能确认时，说“材料不足”，不要编。
- 如果没有命中结果，建议用户先用 FreeRAG 收集相关材料，或换关键词/时间范围。
