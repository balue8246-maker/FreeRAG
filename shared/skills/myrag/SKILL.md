# MyRAG Skill：有限 RAG 深度挖掘

默认使用中文回答、中文整理、中文命名。原始材料本身是英文、代码、表格字段、文件名或专有名词时，保留原文。

FreeRAG 是原材料入口，不是知识库编辑器。MyRAG 的价值不只是“搜到增量”，而是对有限本地语料横过来竖过去读，多视角、多轮次提取，把信息密度榨出来，然后先在当前对话里把“抽离出了什么、哪些可靠、哪些是噪音、下一步能干什么”讲给用户。用户判断后，才决定是否写入 processed、归档、标记 raw 可清理或并入某个项目。

这个方法不是视觉专用，也不是工作专用。FreeRAG 可能收集的是项目讨论、网页、聊天片段、灵感、生活信息、购物比较、错误现场、会议语音、代码、截图、临时想法。MyRAG 要先判断材料和问题属于什么场景，再选择合适的阅读视角。

## 核心原则

- MyRAG 默认自主完成检索、去重、阅读、必要的转写/识图、证据校准和事项化摘要；不要在读了一部分后询问“要不要继续”。只有缺工具/权限/后端、需要外部调研、需要写入长期结果、需要合并项目或需要标记 raw 可清理时，才向用户确认。
- MyRAG 对用户的第一交付物是内容摘要，不是文件审计。文件数量、类型分布和路径是证据和过程信息，不能取代“这批材料里抽出了什么事实、判断、待办、机会和风险”。
- 先检索，再分视角读，不要一眼看完就总结。
- MyRAG 是从碎片和噪音里淘金，不是数垃圾袋；输出重点永远是内容里的事实、判断、数字、待办和机会。
- 每一轮只带一个视角读材料，记录该视角下的证据、例外、矛盾和可迁移结论。
- 区分三件事：原始材料显示、我的整理、我的推断。
- 少材料也能工作，但必须标置信心、证据数量和待确认问题。
- 默认先在当前对话里输出中文 summary，让用户判断下一步；不要默认把 raw 逐条转换成 processed 目录。
- `processed/` 应保存少量高密度结果，例如批次摘要、工作日志、项目结论、代表图和必要原始证据副本；不要把几百个低价值重复 raw 变成几百个 processed 目录。
- 只有当用户确认“这批已经接盘，可以清理 raw”后，才写 `_myrag_done.json` 标记。
- 采样只能用于发现候选和估计分布，不能用来断言“剩余条目全是噪音/都可清理”。凡是要标记清理，必须先枚举明确的 entry id。
- voice 条目标记前必须已有可用 transcript，或用户明确接受 `--force-mark-voice` 的风险；不能把未转写录音当作已接盘。

## 环境能力边界

MyRAG 运行前应判断当前环境是否具备三类能力：

- 文本读取：读取 `content.md`、`llm_context.md`、`transcript.md`、索引和 processed 摘要。
- 图片/截图理解：读取 `image.png`、screen 截图和 storyboard，理解文字、界面状态、视觉层级和表格。
- 音频转写：把 `recording.wav` 转成 `transcript.md`，再抽取内容。

如果图片或音频能力缺失，不要编造内容。先按 `INSTALL_ADAPTERS.md` 配置 Vision / ASR 后端，或明确告诉用户本轮只能处理文本和已有 transcript。

## 淘金输出格式

当用户让你“整理这批语料”“看看最近收了啥”“处理 RAG”“这些截图/聊天里有什么”时，最终必须在当前对话里输出以下内容。不要只输出分类、数量或文件路径。

默认用表格表达，按你判断的信息密度、行动价值和紧急程度排序。最终表格必须是“一个事项一行”：多维度深挖是阅读方法，不是输出结构。不要把同一事项拆成事实表、风险表、方法论表后让用户自己拼回去；要把事实、数字、人物、判断、待办、风险、方法论和证据汇总到同一事项行里。

表格必须有横向分行线，不能只靠竖向列线。优先用当前聊天界面能清楚显示每行边界的表格形式；如果 Markdown 表格只显示列分隔、不显示每行横线，就改用带边框的 HTML `<table>`，或在记录之间加入清晰的分隔行。无论采用哪种渲染方式，仍然必须保持“一个事项一行”。

```markdown
## 我从这批材料里抽出来的东西

### 事项汇总表

| 优先级 | 事项 | 抽出的核心内容 | 关键事实/数字 | 人物/项目 | 判断/风险 | 建议下一步 | 证据 | 置信度 |
| --- | --- | --- | --- | --- | --- | --- | --- | --- |
| P0/P1/P2 | ... | ... | ... | ... | ... | ... | entry/path/截图帧 | high/medium/low |

### 噪音 / 低价值表

| 类型 | 为什么低价值 | 处理建议 |
| --- | --- | --- |
| ... | ... | 跳过/只保留代表/可清理 |

## 我建议下一步
- A. ...（对应事项：...）
- B. ...（对应事项：...）
- C. ...（对应事项：...）

## 证据
- entry/path/截图帧/文件名: 说明支撑了哪条结论。
```

如果材料是电商后台、CRM、数据看板、投放平台、BI、财务/订单/库存等无法 API 接入的界面，优先抽：指标名、数值、时间范围、环比/同比、异常点、筛选条件、账号/店铺/品牌、可行动结论。

如果材料是聊天/会议/讨论，优先抽：谁说了什么、共识、分歧、承诺、deadline、上下游依赖、隐含态度、可复用判断。

如果材料是大量重复截图，先按 SHA-256 或视觉近似去重，只读代表图；重复数量只作为置信度和噪音说明，不是主要结论。

质量要求：

- 精度优先于覆盖。宁可少列事项，也不要把没读深的碎片硬凑成事项。
- 每个事项至少经过 3 个视角检查：事实/数字、人物/项目关系、行动/风险；必要时再加偏好、方法论、反方攻击。
- 最终回答按事项汇总，不按阅读视角拆散。
- 如果某条材料只支持噪音判断，就放进噪音表，不进入事项汇总表。

## 语料协议

FreeRAG 会写入：

- `~/Documents/Corpus/screen/`：屏幕截图、连续采样、storyboard
- `~/Documents/Corpus/clipboard/`：剪贴板文字、图片
- `~/Documents/Corpus/voice/`：录音、转写占位

每个条目通常包含：

- `_meta.json`
- `llm_context.md`
- 原始文件，例如图片、storyboard、音频、Markdown

根目录维护：

- `_library.json`：当前原材料索引，优先读取
- `_index.json`：旧版兼容索引
- `processed/`：LLM 处理后的可复用结果

用户确认需要保存后，处理结果写到：

```text
~/Documents/Corpus/processed/_worklogs/
~/Documents/Corpus/processed/_batches/<batch_id>/
~/Documents/Corpus/processed/<project_or_entry_id>/  # 仅用于确实需要单条或项目级沉淀时
```

推荐文件按场景选择，不要求每条 raw 都生成一套：

- `brief.md`：中文摘要、关键结论、回答当前问题可直接引用的内容
- `deep_read.md`：多视角深读记录，每个视角一节
- `facts.json`：结构化事实、实体、数值、表格、时间线、行动项
- `citations.md`：证据来源，写清原材料文件名、截图帧、时间戳、图片名
- `open_questions.md`：材料不足、互相矛盾、需要用户补充或人工确认的问题
- `ocr.md`：图片、截图、storyboard 中的可读文字、界面状态和视觉观察
- `transcript.md`：语音转写、说话意图、约束、行动项和情绪强度
- `timeline.md`：屏幕多帧、录屏式采样或过程型材料的时间线
- `tables.csv` 或 `tables/*.csv`：截图表格、网页表格、对比表、价格表等结构化数据；字段要保留原始值、来源和置信度
- `digest.md`：一批语料抽离后的对话摘要和下一步建议
- `representatives/`：重复图片、storyboard、音频等少量代表证据副本

当一批或一条 raw 已经完成抽离、代表证据已保留、用户确认可以清理 raw 后，才运行标记：

```bash
python3 scripts/myrag_search.py --mark-processed "<entry_id>" --note "简短说明"
```

这会在原材料目录写入 `_myrag_done.json`。FreeRAG 原材料库的“一键清理已处理过语料”只读取 raw 目录的有效 `_myrag_done.json`，并把 raw 移入 `~/Documents/Corpus/_trash/`，不读取、不解释、不管理 `processed/` 里的处理内容。不要在只是建立索引、去重、创建模板、还没给用户 summary、还没得到用户确认时标记。

标记前必须列出目标 entry id；不要用“我看了样本所以全部标记”。voice 条目如果没有可用 `transcript.md`，脚本会拒绝标记。只有在用户明确知道录音内容已经通过其他方式接盘时，才可使用：

```bash
python3 scripts/myrag_search.py --mark-processed "<voice_entry_id>" --note "用户确认已接盘" --force-mark-voice
```



## 并发处理 / 主脑汇总

MyRAG 遇到多条、多模态、可并行阅读的语料时，应该采用“主会话分派，子 agent 阅读，主会话汇总”的模式。

默认推荐“同一批 RAG，多镜头并发”：主会话先生成代表材料包，然后派 6-10 个只读子 agent 从不同镜头读同一批材料。子 agent 不是按文件分工，而是按问题意识分工；这样能从碎片里挖出更多信息，而不是略读一遍。

常用镜头分派：

| 子 agent | 镜头 | 主要目标 |
| --- | --- | --- |
| A1 | 原始事实/数字 | 抽原话、画面、数值、文件名、时间、指标、金额、数量 |
| A2 | 人物/项目关系 | 抽人、团队、品牌、客户、项目、上下游和责任关系 |
| A3 | 意图/需求 | 抽显性需求、隐含动机、真正要解决的问题 |
| A4 | 决策/取舍 | 抽已经做出的选择、放弃的选项、代价和判断标准 |
| A5 | 行动/机会 | 抽待办、deadline、下一步动作、可推进机会 |
| A6 | 风险/矛盾/缺口 | 抽冲突、证据不足、未确认假设、可能误读 |
| A7 | 模式/复用/SOP | 抽重复出现的方法、流程、模板、prompt 规则 |
| A8 | 情绪/偏好/审美 | 抽用户在意/讨厌/偏好的判断口径和体验标准 |
| A9 | 噪音/去重 | 识别重复、低价值裁切、空白截图、误收材料 |
| A10 | 反方攻击 | 专门质疑主结论，找最容易被攻击的证据和假设 |

不一定每次都派满 10 个。材料少时用 3-5 个镜头；材料多、混合图片/聊天/后台/录音时优先 8-10 个镜头并发。

主会话职责：

- 明确用户问题和本轮处理目标。
- 建立材料清单，按主题、时间段、模态或 entry id 分组。
- 判断哪些工作可以并发，哪些必须等前置结果。
- 给每个子 agent 明确边界：只读哪些 entry、输出什么格式、不要改其他材料。
- 汇总所有子 agent 的观察，做证据校准、反方攻击、交叉合并和最终回答。
- 主会话只能先提出哪些 entry 可标记；给用户看过事项表 summary 并确认 raw 已接盘后，才统一执行 `--mark-processed`。

子 agent 适合承担：

- 图片 / 截图 OCR、视觉观察、表格抽取和 `tables.csv` 草案。
- 屏幕 storyboard 的时间线复盘和状态变化记录。
- 语音转写、语义整理、行动项抽取。
- 某个主题簇、时间段或 entry id 列表的局部深读。
- 去重、低价值噪音识别、候选事项证据整理。

子 agent 输出格式必须便于主脑合并：

```markdown
# 子任务结果

## 范围
- entry id / 路径：
- 模态：text / image / screen / voice / mixed

## 关键观察
- 事项：
- 事实/数字：
- 主张/判断：
- 待办/风险：
- 推断：
- 不确定：

## 结构化产物
- OCR / transcript / timeline / tables.csv 是否建议写回：
- 表格字段或 CSV 草案：

## 证据引用
- 原材料文件、截图帧、音频时间、文本片段：

## 建议状态
- 该镜头下是否足够支撑主脑合并：是 / 否
- 原因：
```

并发规则：

- 可并发就并发，但不要让多个子 agent 写同一个文件；写文件前先分配目录或文件名。
- 图片、屏幕、语音、文本可以分给不同子 agent 先读；最终结论只能由主会话合成。
- 子 agent 只产出当前对话里的证据、观察和草案；如果用户确认保存，优先由主会话统一落盘。
- 子 agent 不负责最终回答用户，也不负责长期项目合并；它们只提供证据和局部判断。
- 子 agent 不写 processed，不标记 `_myrag_done.json`，不移动或删除 raw。
- 子 agent 输出必须以“事项”为锚点，不能只按镜头列散点；主脑要能把不同镜头合并到同一事项行。
- 对 20 条以上或多模态混合语料，默认先并发抽样 / 分簇，再汇总判断是否继续深挖。
- 对大量 clipboard image，先运行 `--image-clusters` 按文件 SHA-256 做精确去重；`count` 是完全相同图片出现次数，不是相似图聚类。主会话只把每簇代表图分给视觉子 agent。
- 主会话的默认交付物是当前聊天里的“抽离总结 + 下一步选项”，不是文件系统里的一堆 processed 目录。

## 多模态处理责任

MyRAG 不是只按 raw 类型列清单。接盘 FreeRAG 原材料后，应主动把不同材料读成可复用处理层：

- 图片 / 截图：做 OCR、视觉观察、界面状态理解；有表格就抽 CSV。
- 屏幕 storyboard：按时间线理解操作过程、状态变化和前后文。
- 音频：先转写，再提取意图、约束、行动项和可引用原话。
- 混合批次：把文本、图片、屏幕、语音一起归拢成候选事项，再深挖。

MyRAG 唯一需要回写给 raw 的信号是 `_myrag_done.json`，意思是“这条原材料已经被 MyRAG 读过并接盘，FreeRAG 可以在用户确认后一键清理 raw”。这一步必须发生在 summary 之后，而不是代替 summary。

## 检索方式

运行：

```bash
python3 scripts/myrag_search.py "<查询关键词或问题>"
```

常用参数：

```bash
python3 scripts/myrag_search.py "<查询>" --format text
python3 scripts/myrag_search.py --entry "<entry_id>" --format text
python3 scripts/myrag_search.py --recent 10 --format text
python3 scripts/myrag_search.py --suggest-projects 40 --format text
python3 scripts/myrag_search.py --image-clusters 40
python3 scripts/myrag_search.py --goldmine-candidates 80
python3 scripts/myrag_search.py "<查询>" --deep-plan --format text
```

脚本随 skill 一起放在 `scripts/myrag_search.py`，调用时优先使用 skill 内的相对路径，避免依赖某台机器上的个人工具目录。

脚本会优先读取 `_library.json`，没有则回退 `_index.json`。检索会看标题、摘要、`llm_context.md` 和已有 `processed/<entry_id>/`，并把 processed 内容加权靠前。

OCR 后端不得写死个人机器路径。脚本只会使用：

- 环境变量 `MYRAG_OCR_COMMAND`，其中可用 `{image}` 和 `{prompt}` 占位。
- 或 PATH 中的 `kimi` / `kimi.js` 命令。

没有可用 OCR 后端时，MyRAG 仍会完成索引、去重和代表图定位，但会明确告诉用户“未配置 OCR 后端”，不假装已经逐字读图。

批处理命令属于高级工具，不是默认工作流：

```bash
python3 scripts/myrag_search.py --process-pending
python3 scripts/myrag_search.py --write-batch-digest "<batch_id>"
python3 scripts/myrag_search.py --compact-batch "<batch_id>"
python3 scripts/myrag_search.py --process-pending --mark-after-process
```

使用限制：

- `--process-pending` 只能作为只读盘点/候选材料辅助，不应取代 LLM 在当前对话里的 summary，也不应逐条写 processed。
- `--mark-after-process` 是兼容旧流程的禁用参数，不作为清理通道；用户确认后仍应对明确 entry id 单独运行 `--mark-processed`。
- 大量重复材料默认应保存批次级 digest 和代表证据，不应在 `processed/` 顶层制造几百个 entry 目录。

用户确认后才使用的写回 / 清理命令：

```bash
python3 scripts/myrag_search.py --init-processed "<entry_id>"
python3 scripts/myrag_search.py --mark-processed "<entry_id>" --note "用户已确认 summary，raw 已接盘"
```

## 默认深挖流程

收到查询后，按这个顺序工作：

1. **明确问题**：把用户问题改写成 1-3 个中文检索意图。不要只照抄关键词。
2. **判断场景**：这是找事实、复盘过程、提取规律、理解偏好、整理待办、做选择、排错，还是找灵感？
3. **召回候选**：用 `myrag_search.py` 搜索，必要时再用 `--recent` 看近期语料。
4. **建立材料清单**：列出命中的条目、类型、路径、已有处理结果情况。
5. **主题/项目归拢**：如果材料很散，先把它们归成几组候选事项，让用户确认这些收束方向是否正确。
6. **选择视角**：从下面的通用视角里选 4-8 个。不要默认全用，也不要只用一个。
7. **分视角深读**：每一轮只回答该视角的问题；先记录观察，不急着总结合并。
8. **证据校准**：把材料里的内容分成事实、主张、愿望、推断、噪音；给关键结论标置信心。
9. **反向审视**：问一句“如果反对这个结论，最容易攻击哪里？”补上矛盾、缺口和反例。
10. **交叉合并**：把多轮观察合并成结论、证据、矛盾、置信度和缺口。
11. **对话交付**：先在当前 Codex / ClaudeCode 对话里直接给用户 summary，包含“抽离出了什么、证据来自哪里、哪些是噪音、哪些不确定、建议下一步”。
12. **等待判断**：让用户决定保存工作日志、并入项目、继续深挖、外部调研、标记 raw 可清理，还是丢弃。
13. **写回处理层**：只有用户确认要保存时，才写 `_worklogs/`、`_batches/` 或项目级 processed；只有用户确认 raw 已接盘时，才对明确 entry id 标记 `_myrag_done.json`。

## 主题/项目归拢

当用户说“整理最近语料”“看一下这些乱七八糟的东西”“这些最终应该收束在哪几件事下”时，先做归拢，不要直接进入深挖。

做法：

1. 读取最近一批语料：

```bash
python3 scripts/myrag_search.py --suggest-projects 40 --format text
```

2. 把条目按语义聚成 3-8 个候选事项。这里的“项目”是宽泛概念，可以是工作项目、产品事项、生活事项、研究主题、审美偏好、待办池、问题线索，不一定是正式项目。
3. 每个候选事项给出：
   - 名称：短中文名。
   - 一句话说明：这坨语料大概在说什么。
   - 代表条目：3-6 个 entry id 或路径。
   - 证据：为什么它们应该归在一起。
   - 置信度：high / medium / low。
   - 建议出口：存工作日志 / 结合已有项目 / 外部调研 / 暂不处理。
4. 最后问用户确认：

```text
我建议先把这些语料收束成以上几件事。你看这些分组对不对？需要改名、合并、拆分或指定某个已有项目吗？
```

不要擅自把候选事项写入长期项目。用户确认前，它们只是“归拢建议”。

归拢时只忽略 FreeRAG 协议/运营层生成物，例如 `_meta.json`、`llm_context.md`、`_index.json`、`_library.json`、`README_FOR_LLM.md`、模板标题和字段名。不要因为语料里出现 `FreeRAG`、`LLM`、`RAG`、某个 app 名、公司名、项目名或技术名就过滤掉；这些词可能正是用户真实讨论的主题。

## 收束出口

MyRAG 的终点不是“继续无限分析”，而是把本地 RAG 挖透后给出明确出口：

### 1. 存工作日志

适用情况：

- 这次深挖本身已经形成了有价值的过程记录、结论、偏好、决策或待办。
- 暂时不需要外部信息，也不需要挂到某个已有项目。
- 用户只是想把这次思考留住，方便以后找回。

做法：

- 询问是否保存工作日志；如果用户同意，生成中文 Markdown。
- 默认写入 `~/Documents/Corpus/processed/_worklogs/`。
- 文件名使用 `yyyyMMdd_HHmm_<中文主题>.md`。
- 内容包括：问题、命中材料、深挖视角、关键结论、证据、推断、待确认、下一步。

### 2. 结合已有项目

适用情况：

- 深挖结果明显应该进入某个项目、产品、客户、研究主题、个人长期事项。
- 单独保存日志不够，需要和项目既有信息合并。

做法：

- 不要猜项目。先问用户：要结合哪个项目？项目资料路径在哪里？
- 用户给出项目后，再读取项目资料，判断哪些结论能合并、哪些只是参考、哪些冲突。
- 输出项目侧可用的增量材料，例如项目日志、决策备忘、规则更新、待办清单或风险列表。

### 3. 需要外部调研

适用情况：

- 本地 RAG 只能说明“我已有材料里有什么”，但问题需要最新事实、市场信息、竞品状态、政策、价格、版本、公开资料或第三方证据。
- 材料里有主张但缺验证。
- 反方攻击点主要来自外部事实缺口。

做法：

- 明确说明本地 RAG 已经挖到哪里、还缺什么外部信息。
- 如果用户问题本身需要当前外部事实，直接进入 websearch 调研。
- 调研后要把外部事实和本地 RAG 分开标注，不要混在一起。

如果三类都不适用，就直接回答用户并停止。不要为了显得勤奋而继续扩展。

## 通用视角系统

根据任务选择，不要机械全用：

- `L01 原始事实镜头`：材料明确出现了什么？原话、画面、数值、文件、链接是什么？
- `L02 时间与情境镜头`：这发生在什么时间、什么场景、什么前后文里？
- `L03 人事物关系镜头`：涉及哪些人、项目、产品、地点、账号、文件？彼此是什么关系？
- `L04 意图与需求镜头`：显性需求是什么？隐含动机、真正想解决的问题是什么？
- `L05 结构与机制镜头`：材料内部的层级、流程、因果、约束、规则、依赖是什么？
- `L06 细节与数值镜头`：容易漏掉但重要的细节、参数、版本、价格、数量、字段、阈值是什么？
- `L07 情绪与偏好镜头`：用户在意什么、讨厌什么、审美/口味/工作习惯/判断标准是什么？
- `L08 矛盾与缺口镜头`：哪些地方互相冲突、证据不足、说法含糊、需要补材料？
- `L09 模式与反例镜头`：多条材料中反复出现什么模式？有什么例外或反例？
- `L10 决策与取舍镜头`：已经做了什么选择？为什么放弃其他选项？代价是什么？
- `L11 行动与机会镜头`：接下来能做什么？谁做、怎么做、做到什么算完成？
- `L12 沉淀与复用镜头`：哪些内容值得变成长期记忆、SOP、偏好、模板、检查清单或 prompt 规则？
- `L13 反方攻击镜头`：如果有人不同意这个结论，会攻击哪条证据、哪个假设、哪个遗漏？

### 场景到视角的选择

按问题类型动态选视角：

- **找事实/找出处**：L01、L02、L03、L06、L08、L09
- **复盘一件事**：L02、L04、L05、L08、L10、L11、L13
- **做决策/比较方案**：L04、L05、L06、L08、L10、L11、L13
- **提取用户偏好**：L04、L07、L08、L09、L12
- **整理项目记忆**：L02、L03、L05、L10、L11、L12
- **处理生活/消费/旅行等碎片**：L01、L02、L04、L06、L07、L11
- **排错/定位问题**：L01、L02、L05、L06、L08、L11、L13
- **创意/审美/视觉材料**：先用通用视角，再按需要加画面、风格、版式、信息密度等专项视角。

## 深挖不是套模板

不要把上面的视角当固定表格填空。正确做法是：

1. 先快速读一遍候选材料，判断“这批材料可能有价值的层次”。
2. 选少数最相关视角逐轮读。
3. 如果某一轮读出新问题，再临时加视角。
4. 如果某个视角明显无价值，跳过。
5. 最终输出要合并成用户能用的结论，不要把过程文档原样塞给用户。

## 证据校准

对关键结论做四类标注，尤其是用户要决策、复盘、比较、判断趋势时：

- **可验证事实**：材料里明确出现，能指向原文、截图帧、录音转写、文件或数值。
- **材料主张**：材料里有人这么说，但还缺外部验证或反例检查。
- **意图/愿望**：想做、计划做、希望做到，不等于已经做到。
- **我的推断**：基于材料合理推出，但不是材料直接给出的事实。

不要混淆“已经发生”和“想要发生”。不要把用户或材料里的表达自动当事实。没有证据就标“材料不足”。

## 深挖质量门槛

- 结论先行，但不能只有骨架；关键判断要有足够上下文和证据。
- 能表格化时表格化，方便比较，但不要为了表格牺牲细节。
- 每个重要判断至少给出来源或说明为什么只是推断。
- 对多路径问题，给出路径、前提、触发信号、风险，而不是只给单一路线。
- 最后用很短一段写“还值得继续深挖什么”，只列真正的信息缺口。

## 按材料类型处理

本节定义运行时的通用处理链路。具体模型、Vision 后端、ASR 后端和安装命令放在 `INSTALL_ADAPTERS.md`，不要把个人机器路径写进这里。

### 剪贴板文字 `clipboard/text`

- 先读 `content.md` 原文，保留关键句、代码、表格、链接、错误信息和专有名词。
- 如果多条 clipboard 在短时间内连续捕获、内容互补，先判断是否是同一长文本被切割；能还原就按时间顺序拼回主题，不要把单字/半句当成独立事项。
- 完整长文本先切结构：主题、论点、证据、定义、待办、引用、来源和不确定点。
- 对用户交付时，把多条相关 clipboard 合并成一个事项；孤立低价值碎片放入噪音表。
- 用户确认后，有长期价值才写入 `_worklogs/`、`_batches/` 或项目目录；无长期价值只在确认后标记 raw 可清理。

### 剪贴板图片 `clipboard/image`

- 大量图片先运行 `--image-clusters`，按 SHA-256 精确折叠完全重复图，只读每簇代表图；`count` 只作为重复强度和置信度背景。
- 识图时不只 OCR，还要读界面状态、视觉层级、空间关系、上下文和用户当时可能在做什么。
- 聊天截图要抽谁和谁在讨论什么、共识/分歧/承诺/隐含态度；报错截图要保留原始错误信息；后台/看板截图要抽指标、时间范围、筛选条件、账号和异常点。
- 如果图里有表格、报价、参数、排期、清单、对比矩阵，必须抽成 CSV 草案；字段尽量包含 `source`、`table_name`、`row_index`、原始列名、`confidence`、`note`。
- 用户确认保存后，按需要写 `ocr.md`、`tables.csv` 或 `tables/*.csv`、`representatives/`。

### 屏幕截图 `screen`

- 先检查目录是否真的有有效截图或 storyboard；空目录、失败捕获和无内容帧应进入噪音表。
- 单张截图按图片规则处理；连续采样和 storyboard 要按时间顺序读，再横向比较状态变化。
- 输出时说明用户当时在做什么、界面暴露了什么状态、前后帧有什么变化、哪些证据支持结论。
- 多帧过程型材料适合写 `timeline.md`；有表格写 `tables.csv`；有界面文字和视觉状态写 `ocr.md`。

### 语音 `voice`

- 先检查是否已有可用 `transcript.md`。如果已有完整转写，直接读转写并抽取内容。
- 如果只有 `recording.wav` 和待转写占位，应先尝试 ASR；没有 ASR 后端时明确说明无法转写，不要假装读过音频。
- 语音摘要不要输出逐字稿，也不要三行高度浓缩。长会议应按话题组织，提取关键发言、数字、判断、行动项、分歧、风险、时间点和待确认问题。
- 如果需要区分说话人，先完成转写，再使用 diarization 后端生成带说话人标签的 `transcript_labeled.md`；没有后端时用“说话人未区分”标注。
- 转写全文可以留在 raw voice 条目目录；结构化摘要只有用户确认后才写 `_worklogs/` 或项目目录。

### 空目录 / 低价值条目

以下内容不进入事项汇总，只放噪音表或跳过：

- 目录存在但无任何有效文件。
- 单字、单标点、无法还原上下文的剪贴板碎片。
- 完全重复内容，保留代表即可。
- FreeRAG 协议/运营层生成物本身，例如 `_meta.json`、`llm_context.md`、`_index.json`、`_library.json`、`README_FOR_LLM.md`。这些可作索引和元信息，不应被当作用户内容主题。

### 处理链路总览

```text
收到语料
  ├─ 剪贴板文字 → 判断密度 → 碎片拼接或单条阅读 → 抽关键信息
  ├─ 剪贴板图片 → SHA-256 去重 → Vision/OCR → 抽文字、表格、界面状态
  ├─ 屏幕截图   → 检查是否空目录 → 按帧/按图阅读 → timeline / OCR / CSV 草案
  ├─ 语音       → 检查 transcript → 必要时 ASR → 结构化摘要
  └─ 低价值条目 → 噪音表，不进事项表

最终输出：先在当前对话里给用户内容摘要；用户确认后才落盘或标记 raw 可清理。
```
